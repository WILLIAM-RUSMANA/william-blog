The final technical phase of a pentest is producing a report. This is arguably the most important deliverable — it is what the client actually receives and acts on.

Three Report Types:

Executive Report — for senior leadership (non-technical):


Project objective and scope summary
Vulnerability risk classification overview
Executive summary of key findings
Statistics and a risk matrix
Focused on business impact, not technical detail


Management Report — for IT managers and compliance teams:


Compliance achievement status
Testing methodology overview
Assumptions and limitations of the test
Change management and configuration management implications


Technical Report — for the security and IT teams who will actually fix things:


Detailed security issues with evidence
Vulnerabilities map — which systems are affected by what
Exploits map — what was successfully exploited
Best practices and remediation recommendations


Sample Report Structure:
Legal notice → Pentest agreement → Introduction → Project objective → Assumptions & limitations → Vulnerability risk scale → Executive summary → Risk matrix → Testing methodology → Security threats → Recommendations → Vulnerabilities map → Exploits map → Compliance assessment → Change management → Best practices → Annexes

Post-Testing Procedures — after reporting, the tester should ensure all backdoors are removed, systems are restored to their pre-test state, and a debrief is held with the client.
